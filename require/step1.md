# 專案題目

多租戶課程與預約平台（商家可開課、學員可預約、可付款、可候補、可訂閱）

## 1) 系統範圍（Scope）

系統提供：
1. 多租戶（每個商家一個租戶）
2. 前台（學員）與後台（商家/平台管理員）
3. 課程與時段管理
4. 預約、取消、候補、改期
5. 金流與退款
6. 通知（站內、Email、Webhook）
7. 報表與結算
8. API 與第三方整合

不包含：
1. 實體門禁硬體控制
2. 視訊串流服務本體（僅整合外部服務）
3. ERP 全套功能

## 2) 角色（Actors）

1. 訪客：未登入使用者
2. 學員（Member）
3. 講師（Instructor）
4. 商家管理員（Tenant Admin）
5. 商家客服（Tenant Staff）
6. 平台管理員（Platform Admin）
7. 付款閘道（Payment Gateway，外部系統）
8. 通知服務（Email/SMS/Push/Webhook，外部系統）
9. 排程與背景工作者（Scheduler/Queue Worker，系統角色）

## 3) 主要領域物件（Domain Entities）

1. Tenant（租戶）
2. User（使用者）
3. Role / Permission（權限）
4. Course（課程）
5. Session（課程場次）
6. Booking（預約）
7. Waitlist（候補）
8. Plan / Subscription（訂閱方案）
9. Order / Payment / Refund（訂單、付款、退款）
10. Coupon / Promotion（優惠）
11. Notification（通知）
12. Attendance（出席）
13. Invoice（發票）
14. Settlement（商家結算）
15. Audit Log（稽核紀錄）

## 5) 詳細 Use Case 規格（核心 20 個）

格式：
- 主要角色
- 前置條件
- 觸發條件
- 主流程
- 替代流程 / 例外流程
- 後置條件
- 商業規則

### UC-01 訪客註冊

- 主要角色：訪客
- 前置條件：無
- 觸發條件：點擊註冊
- 主流程：
1. 訪客輸入 Email、密碼、姓名
2. 系統驗證欄位與密碼強度
3. 系統建立 user（未驗證）
4. 系統寄送驗證信
- 例外流程：
1. Email 已存在 -> 顯示已註冊並可改走登入/忘記密碼
- 後置條件：帳號建立成功，等待驗證
- 商業規則：同 Email 不可重複；密碼需符合安全策略

### UC-02 Email 驗證

- 主要角色：學員
- 前置條件：已註冊未驗證
- 觸發條件：點驗證連結
- 主流程：
1. 系統驗證 token 與期限
2. 更新 Email verified_at
3. 導向登入或首頁
- 例外流程：
1. token 過期 -> 可重寄
- 後置條件：帳號完成驗證

### UC-03 使用者登入

- 主要角色：學員/員工/管理員
- 前置條件：帳號存在、啟用中
- 觸發條件：送出登入表單
- 主流程：
1. 驗證帳密
2. 若啟用 2FA -> 驗證 OTP
3. 建立 session/token
- 例外流程：
1. 密碼錯誤超限 -> 暫時鎖定
2. 帳號停用 -> 拒絕登入
- 後置條件：登入成功

### UC-04 平台建立租戶

- 主要角色：平台管理員
- 前置條件：平台管理員已登入
- 觸發條件：建立新商家
- 主流程：
1. 輸入租戶名稱、方案、網域識別
2. 建立 tenant、預設角色、預設設定
3. 指定首位 tenant admin
- 例外流程：
1. 子網域衝突 -> 要求更換
- 後置條件：租戶可啟用使用

### UC-05 租戶邀請員工

- 主要角色：商家管理員
- 前置條件：租戶已啟用
- 觸發條件：發送邀請
- 主流程：
1. 輸入員工 Email 與角色
2. 系統建立 invitation
3. 系統發送邀請連結
4. 員工接受邀請建立帳號/綁定帳號
- 例外流程：
1. 邀請過期 -> 可重發
- 後置條件：員工加入租戶

### UC-06 建立課程模板

- 主要角色：商家管理員/講師
- 前置條件：有課程管理權限
- 觸發條件：新增課程
- 主流程：
1. 輸入名稱、描述、分類、價格
2. 設定人數、取消政策、候補政策
3. 儲存為 draft
4. 可發佈為公開
- 例外流程：
1. 權限不足 -> 拒絕
- 後置條件：課程模板可供排課

### UC-07 建立循環場次

- 主要角色：商家管理員/講師
- 前置條件：課程模板存在
- 觸發條件：建立排程
- 主流程：
1. 選擇週期（如每週二 19:00）
2. 設定起訖日期、例外日期
3. 系統批次建立 sessions
- 例外流程：
1. 講師時段衝突 -> 系統提示衝突清單
- 後置條件：場次建立完成

### UC-08 學員預約場次

- 主要角色：學員
- 前置條件：已登入、帳號有效、可預約
- 觸發條件：點擊預約
- 主流程：
1. 系統檢查容量、截止時間、資格
2. 若需付費，建立訂單並導流付款
3. 付款成功後建立 booking（confirmed）
4. 發送成功通知
- 例外流程：
1. 滿額 -> 提供候補
2. 付款失敗 -> booking 不成立或 pending
- 後置條件：預約成立或候補中

### UC-09 候補加入

- 主要角色：學員
- 前置條件：場次已滿
- 觸發條件：點擊加入候補
- 主流程：
1. 建立 waitlist 項目（依規則排序）
2. 釋出名額時依序通知第一順位
3. 給定回覆時限
4. 逾時則遞補下一位
- 例外流程：
1. 候補上限已滿 -> 拒絕加入
- 後置條件：候補成功或轉正成功

### UC-10 學員取消預約

- 主要角色：學員
- 前置條件：booking 狀態為 confirmed
- 觸發條件：點擊取消
- 主流程：
1. 系統判斷是否在可免費取消期限內
2. 更新 booking=cancelled
3. 觸發退款或扣罰規則
4. 釋放名額並觸發候補遞補
- 例外流程：
1. 超過取消時限 -> 不可取消或需手續費
- 後置條件：預約取消完成

### UC-11 客服代客改期

- 主要角色：商家客服
- 前置條件：有客服權限、原 booking 有效
- 觸發條件：客服操作改期
- 主流程：
1. 選擇新場次
2. 驗證新場次可預約
3. 取消舊 booking 並建立新 booking（原價轉移）
4. 留存稽核紀錄
- 例外流程：
1. 新場次已滿 -> 拒絕或轉候補
- 後置條件：改期完成

### UC-12 付款成功回寫

- 主要角色：付款閘道（外部）
- 前置條件：訂單 pending
- 觸發條件：Webhook 回傳成功
- 主流程：
1. 驗證 webhook 簽章與冪等鍵
2. 更新 payment=paid、order=completed
3. 若為預約訂單則確定 booking
4. 發送付款成功通知
- 例外流程：
1. 重複 webhook -> 忽略（冪等）
- 後置條件：交易狀態一致

### UC-13 退款處理

- 主要角色：客服/管理員
- 前置條件：交易已付款
- 觸發條件：發起退款
- 主流程：
1. 選擇全額或部分退款
2. 呼叫 payment gateway API
3. 更新 refund 狀態與 booking/order 狀態
4. 通知會員
- 例外流程：
1. gateway 失敗 -> 記錄並進入重試佇列
- 後置條件：退款成功或待重試

### UC-14 訂閱續約

- 主要角色：排程系統 + 付款閘道
- 前置條件：subscription=active、到期日將至
- 觸發條件：排程每日掃描
- 主流程：
1. 建立續約訂單
2. 自動扣款
3. 成功則延長週期
4. 失敗則進入 dunning 流程（提醒/重試）
- 例外流程：
1. 連續失敗超限 -> subscription past_due/cancelled
- 後置條件：訂閱更新

### UC-15 開課提醒通知

- 主要角色：排程系統
- 前置條件：場次即將開始
- 觸發條件：T-24h / T-1h 觸發排程
- 主流程：
1. 查詢符合條件 bookings
2. 送出站內 + Email
3. 記錄發送狀態
- 例外流程：
1. 通知服務故障 -> 佇列重試
- 後置條件：提醒已發送或待重試

### UC-16 簽到與出席

- 主要角色：講師/客服
- 前置條件：場次已開始
- 觸發條件：點名操作
- 主流程：
1. 開啟場次名單
2. 標記出席/遲到/缺席
3. 套用點數或懲罰規則
- 例外流程：
1. 場次未開始且禁止提前點名 -> 拒絕
- 後置條件：attendance 完成

### UC-17 營收報表產生

- 主要角色：商家管理員
- 前置條件：具備報表權限
- 觸發條件：指定期間產報表
- 主流程：
1. 聚合訂單、退款、稅額、淨收入
2. 產生可下載報表
3. 可排程定時寄送
- 例外流程：
1. 大量資料 -> 改走背景任務完成後通知
- 後置條件：報表產出

### UC-18 結算單產生

- 主要角色：平台管理員
- 前置條件：租戶已有交易
- 觸發條件：月底結算排程
- 主流程：
1. 計算平台抽成、金流手續費、應撥款
2. 產生 settlement 記錄
3. 匯出對帳檔
- 例外流程：
1. 異常交易未對齊 -> 標記待人工覆核
- 後置條件：結算單建立

### UC-19 稽核日誌查詢

- 主要角色：平台管理員/租戶管理員（受限）
- 前置條件：具稽核權限
- 觸發條件：查詢稽核頁
- 主流程：
1. 依操作者、時間、事件類型篩選
2. 查看關鍵操作（退款、權限變更、代客操作）
- 例外流程：
1. 權限不足 -> 僅可見自租戶且脫敏資料
- 後置條件：合規可追溯

### UC-20 API 建立預約

- 主要角色：第三方系統
- 前置條件：持有效 API token
- 觸發條件：呼叫 POST /bookings
- 主流程：
1. 驗證 token scope 與 tenant
2. 驗證 session 可預約
3. 建立 booking
4. 回傳 booking id 與狀態
- 例外流程：
1. 429 超頻 -> 拒絕並回 retry-after
2. 重複請求（冪等鍵）-> 回傳既有結果
- 後置條件：API 預約完成

## 6) 全域商業規則（Business Rules）

1. 同一學員不可重複預約同一場次
2. 預約截止時間後不可新預約
3. 取消是否退款由課程取消政策決定
4. 候補遞補需限時確認，逾時順位後移
5. 所有金流 webhook 必須簽章驗證且冪等
6. 多租戶資料嚴格隔離（tenant_id）
7. 高風險操作必須寫稽核日誌
8. 角色權限採最小權限原則
9. 報表口徑需可追溯至交易明細
10. 個資匯出/刪除需留合規紀錄

## 7) 非功能需求（NFR）

1. 可用性：99.9%（核心預約與付款）
2. 效能：95% API < 300ms（不含第三方支付）
3. 一致性：付款、訂單、預約狀態最終一致且可補償
4. 安全：OWASP Top 10 基本防護、2FA、加密敏感資料
5. 擴展性：支援橫向擴充 queue / cache / app server
6. 可觀測性：集中 log、追蹤 ID、告警
7. 可測試性：核心流程具整合測試與契約測試
